# RAIS-RFC-0002: Stream Metadata Event

- **Number:** 0002
- **Title:** Stream Metadata Event
- **Status:** Draft
- **Author:** trimooo (GitHub: @trimooo)
- **Created:** 2026-05-14
- **Target version:** RAIS v1.1 (minor — additive only)

---

## Summary

Add an optional `metadata` event type that servers may emit alongside or after the `done` event to communicate stream-level statistics and model information. Unlike RFC-0001 (tool calls), this is purely additive and non-breaking enough to target a minor version (v1.1).

---

## Motivation

After a stream completes, clients often want to know:

- Which model produced the response (for multi-model comparisons)
- How many tokens were used (for cost tracking)
- How long generation took (for performance monitoring)
- The finish reason (completed naturally vs. hit max tokens)

Currently, none of this information is available in the RAIS wire format. Clients either make a second API call, parse proprietary response headers, or remain blind to these metrics.

The `metadata` event enables:

- **Token cost tracking**: Show users their token usage in real-time or per-response
- **Model attribution**: Display "Generated by claude-sonnet-4-6 · 342 tokens" in the UI
- **Performance monitoring**: Feed data to DevTools, analytics, and observability pipelines
- **Finish reason handling**: Distinguish "max tokens reached" from "natural completion" to prompt users to continue

---

## Detailed design

### Wire format

```json
{
  "type": "metadata",
  "model": "claude-sonnet-4-6",
  "tokens": {
    "input": 128,
    "output": 342,
    "total": 470
  },
  "elapsed_ms": 1840,
  "finish_reason": "stop"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | `"metadata"` | yes | Discriminant |
| `model` | `string` | no | Model identifier as returned by the LLM provider. |
| `tokens` | `object` | no | Token usage. All sub-fields are optional. |
| `tokens.input` | `number` | no | Input / prompt tokens consumed. |
| `tokens.output` | `number` | no | Output / completion tokens generated. |
| `tokens.total` | `number` | no | Total tokens (input + output). May differ from sum on some providers. |
| `elapsed_ms` | `number` | no | Wall-clock time from first token to stream end, in milliseconds. |
| `finish_reason` | `string` | no | Why the stream ended. Suggested values: `"stop"` (natural end), `"length"` (max tokens), `"error"` (corresponds to an `error` event), `"abort"` (client cancelled). |

All fields except `type` are optional. A server that only knows the model name can emit:
```json
{"type": "metadata", "model": "gpt-4o"}
```

### Event sequencing

`metadata` MUST be emitted **after** `done` or `error`:

```
data: {"type":"text","text":"Hello!"}

data: {"type":"done"}

data: {"type":"metadata","model":"claude-sonnet-4-6","tokens":{"output":2},"elapsed_ms":412}

```

Alternatively, `metadata` MAY be emitted as part of a single event that combines with `done`:

```
data: {"type":"done","metadata":{"model":"...","tokens":{...}}}
```

> **Note:** This alternative embeds metadata into the `done` event as an optional `metadata` field. This RFC proposes both options and asks reviewers to choose.

### Client behavior

- Clients MUST NOT require `metadata` to be present — it is always optional
- Clients that receive `metadata` SHOULD surface relevant fields to the user or developer tools
- Clients MUST accept `metadata` after `done` and MUST NOT treat it as an error
- Clients MUST silently ignore unknown fields within `metadata`

### Server behavior

- Servers SHOULD emit `metadata` only after `done` or `error`
- Servers MUST NOT emit `metadata` mid-stream before content tokens (this would confuse v1 clients' stream-end detection)
- Servers SHOULD include at minimum the `model` field if known

---

## Backward compatibility

### Existing v1 clients

v1 clients already MUST silently ignore unrecognized event types (SPEC.md §8). A `metadata` event after `done` is ignored cleanly — the stream is already considered complete after `done`. No behavior change.

### Existing v1 servers

v1 servers simply don't emit `metadata`. No migration required. They remain compliant.

This is why this RFC targets v1.1 rather than v2 — the change is purely additive with no impact on existing implementations.

---

## Alternatives considered

### HTTP response trailers

Metadata could be delivered as HTTP trailers (key-value pairs appended after the body). Rejected because trailers are not reliably supported in all HTTP/1.1 clients and proxies, and would require a separate parsing code path.

### Response headers

Metadata could be returned as HTTP response headers (already in the response before body starts, so only available for post-stream data via a second request). Rejected because it requires a second round-trip and adds state to the connection.

### Embedding metadata in the `done` event

`{"type":"done","model":"...","tokens":{...}}` — no new event type needed. This is the simplest design and avoids adding a new event type to the protocol. The downside is it complicates the `done` event shape and may require v1 clients to be updated to ignore new fields on a known event type (though spec §8 already handles this via "silently ignore unknown fields").

This option is worth serious consideration and may be preferred over a separate `metadata` event.

---

## Security considerations

- `model` field reveals which LLM is in use. Servers that want to keep their model choice private SHOULD omit this field.
- Token counts may indirectly reveal information about the system prompt length (via `tokens.input`). Servers that use private system prompts SHOULD omit `tokens.input` or `tokens.total`.

---

## Open questions

1. **Separate event or field on `done`?** The two options (separate `metadata` event vs. `{"type":"done","metadata":{...}}`) have different implications for parsing. Which is cleaner?
2. **Should `finish_reason` use an enum or be a free string?** An enum provides client-side guarantees but limits extensibility. A free string is extensible but requires clients to handle unknown values gracefully.
3. **Should `elapsed_ms` be measured from stream open or from first token?** Both are useful for different purposes.
